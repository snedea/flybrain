## Phase 5: Spec Compliance and Behavioral Enrichment

- [x] T5.1: Implement graded wind response with brace and orient behaviors per spec -- The spec says wind (air tool) should cause "Brace, orient, or take off" but currently all wind strength feeds uniformly into MECH_JO which accumulates fear, producing only startle/fly escape at any intensity. Three changes needed: (1) Add wind-direction tracking: compute wind angle from the air tool drag vector (dragStart to currentMousePos, already tracked) and store as BRAIN.stimulate.windDirection in handleCanvasMousemove and handleCanvasMouseup so the connectome knows which direction wind is coming from, (2) Add a "brace" behavior state for weak-to-moderate wind: when BRAIN.stimulate.wind is true and windStrength is below ~0.5 and accumStartle has not reached the startle threshold, the fly should decelerate to near-zero speed and orient to face into the wind (targetDir = windDirection + PI, facing the wind source); add brace to BEHAVIOR_MIN_DURATION (500ms), BEHAVIOR_COOLDOWN (1000ms), and insert it into evaluateBehaviorEntry between groom and rest in priority with entry condition checking wind active and windStrength below threshold and not cooling down; in computeMovementForBehavior, brace sets targetSpeed=0 and targetDir toward wind source; add brace to syncBrainFlags as non-moving, (3) Add brace animation: in drawLegs during brace state, widen leg stance slightly (increase base hip angles by ~10%) and suppress jitter (multiply jitter amplitude by 0.1) to show the fly bracing; in drawAntennae during brace state or any active wind, bias antenna target angles toward the wind direction for a wind-sensing posture, (4) Scale MECH_JO stimulation by windStrength in connectome.js (currently BRAIN.stimulate.wind triggers full MECH_JO activation regardless of windStrength at connectome.js:310-313; change to dendriteAccumulateScaled with windStrength so weak wind produces less fear accumulation and strong wind still triggers startle/fly). Files: js/main.js, js/connectome.js. [SPI-]

- [x] T5.2: Implement dark settling behavior and light-responsive activity modulation per spec -- The spec says light changes cause "Phototaxis (move toward light), or settle in dark" but the "settle in dark" half is unimplemented; currently switching to Dark mode only prevents phototaxis entry (lightLevel < 0.5 threshold) without actively promoting settling. Three changes needed: (1) Add light-level-dependent drive modulation in BRAIN.updateDrives at connectome.js: when lightLevel is low (< 0.3), increase fatigue gain rate (e.g. fatigue += 0.006 instead of 0.003 per tick when moving, representing the fly winding down in darkness) and decrease curiosity bias (reduce the random walk range from 0.03 to 0.01, making the fly less exploratory in the dark); when lightLevel is 0, also reduce the tonic background activity injection in the central complex (CX_FC, CX_EPG, CX_PFN at connectome.js:367-371) from 8 to 4, lowering baseline locomotion drive so the fly naturally gravitates toward idle/rest states, (2) Lower the rest fatigue threshold in darkness: in evaluateBehaviorEntry, when lightLevel is 0, use a reduced restFatigue threshold (e.g. 0.4 instead of 0.7) so the fly enters rest state much sooner in the dark, producing the "settle" behavior the spec describes, (3) Add subtle dark-mode animation changes: in drawAntennae, when lightLevel is 0, double the antenna twitch interval (slower, sleepier twitching) by scaling anim.antennaNextInterval by 2; in drawLegs, when lightLevel is 0, reduce idle leg jitter intensity by 50%; these visual cues reinforce the settling behavior. Files: js/connectome.js, js/main.js. [-PI-]

- [x] T5.3: Wire dormant connectome pathways to user interactions or remove dead infrastructure -- Six sensory neuron types in constants.js have full connectome wiring (weights, downstream targets) but zero user-facing activation, meaning ~30% of the sensory connectome is dead code: temperature (THERMO_WARM/THERMO_COOL with mutual inhibition and connections to CX_HDELTA, LH_AV, LH_APP, drives), water (GUS_GRN_WATER to SEZ_WATER to MN_PROBOSCIS), danger odor (OLF_ORN_DANGER to LH_AV/MB_MBON_AV/DN_STARTLE/DRIVE_FEAR), bitter taste (GUS_GRN_BITTER to LH_AV/SEZ_FEED inhibition), nociception (NOCI to DN_STARTLE/DRIVE_FEAR/DN_FLIGHT), and lightDirection (declared at connectome.js:143 but never set or read). Additionally BRAIN.stimulate.dangerOdor and BRAIN.stimulate.temperature exist but nothing sets them. Wire the most impactful pathways: (1) Add a temperature control to the toolbar (a simple cycle button like light: "Temp: Warm/Cool/Neutral") that sets BRAIN.stimulate.temperature to 0.75/0.25/0.5; the existing connectome.js temperature processing at lines 339-347 already handles these values correctly so only the UI button, toolbar wiring, and initial state need to be added; warm should make the fly more active and avoidant, cool should make it exploratory and approach-seeking per the THERMO_WARM/COOL weights, (2) Wire nociception to repeated rapid touches: if the user touches the fly 3+ times within 4 seconds (track touch timestamps in an array), trigger BRAIN.stimulate.nociception = true for one brain tick, activating the NOCI pathway (DN_STARTLE weight 10, DRIVE_FEAR weight 8, DN_FLIGHT weight 6) for an intense escape response distinct from single-touch startle; add nociception stimulus processing in the sensory block of connectome.js BRAIN.update, (3) Remove lightDirection property from BRAIN.stimulate since directional light is not implemented (phototaxis uses canvas center) and the field is misleading dead state, (4) Leave GUS_GRN_WATER, GUS_GRN_BITTER, and OLF_ORN_DANGER in the connectome weights as dormant infrastructure for future expansion but add a code comment noting they are not yet wired to user interactions. Files: js/main.js, js/connectome.js, index.html, css/main.css. [-PI-]

- [x] T5.4: Add automated tests for core simulation logic -- Zero test coverage exists for any of the core systems (connectome propagation, drive updates, behavior state machine, angle math). Add a minimal but meaningful test suite that can run without a build step: (1) Create a test runner: add a tests/run.html page that loads constants.js, connectome.js, and a new tests/tests.js file, runs all test functions, and displays pass/fail results in the page with a summary count; use a simple assertion helper (assertEqual, assertTrue, assertClose for floats) rather than importing a test framework, keeping the no-dependencies constraint, (2) Connectome signal propagation tests: verify that calling BRAIN.setup() initializes all neuron states, that BRAIN.dendriteAccumulate('VIS_R1R6') propagates signal to its postSynaptic targets (VIS_ME, VIS_LO per constants.js weights), that BRAIN.fireNeuron cascades correctly, and that motor neuron readMotor drains state to zero after reading, (3) Drive system tests: verify that BRAIN.updateDrives() increases hunger by 0.005 per tick when not feeding, decreases hunger by 0.3 when _isFeeding is true, increases fear by 0.3 when touch stimulus is active, applies 0.85 exponential decay to fear, and clamps all drives to [0, 1], (4) Angle normalization tests: verify normalizeAngle returns values in [-PI, PI] for inputs like 0, PI, -PI, 3*PI, -5*PI, and that the result is correct for each, (5) Behavior threshold tests: verify evaluateBehaviorEntry returns correct states for known accumulator/drive combinations (e.g. accumStartle=35 with no cooldown returns 'startle', accumFeed=10 with nearby food returns 'feed', high fatigue returns 'rest'). Files: tests/run.html, tests/tests.js (new files). [-PI-]

## Discovery Round 16

- [x] D16.1: Fix test runner silent failure swallowing and hardened test infrastructure -- The test runner (runAllTests in tests/tests.js) has a critical bug: it catches all TestFailure exceptions in a try/catch and writes results only to DOM elements via innerHTML. When run in Node.js (the only automated test environment), failed tests produce exit code 0 with zero console output, making automated regression detection completely non-functional. Confirmed by injecting a deliberately failing test: exits 0 silently. Three related issues to fix together: (1) Add a CLI-compatible output path to runAllTests: after the test loop, console.log the pass/fail/total counts and each failure message, then throw or set process.exitCode = 1 when failures > 0 so Node.js execution surfaces regressions; guard the console/process calls behind typeof checks so the browser path is unaffected, (2) The five functions copied verbatim from main.js into tests.js (normalizeAngle, BEHAVIOR_THRESHOLDS, isCoolingDown, hasNearbyFood, evaluateBehaviorEntry) are frozen snapshots that will silently diverge when main.js changes, causing tests to validate stale logic while the real code regresses. Refactor main.js to separate pure testable functions and constants from DOM-dependent initialization into a shared file (e.g. js/fly-logic.js loaded before main.js) that both main.js and tests.js can reference, eliminating the copy-paste drift risk, (3) Add test coverage for T5.1-T5.3 features that currently have zero tests: dark settling drive modulation (fatigue gain doubles to 0.006 at lightLevel < 0.3, curiosity range drops to 0.02, tonic injection halves to 4 at lightLevel 0), dark rest threshold (0.4 vs 0.7), temperature stimulus routing (THERMO_WARM fires at temperature > 0.65, THERMO_COOL at < 0.35), nociception single-tick auto-clear behavior, and brace entry conditions (wind active AND windStrength < 0.5 AND accumStartle below threshold). Also fix the misleading windDirection comment at connectome.js:142 which says "direction wind is blowing FROM" but the code computes and uses it as the wind travel direction (drag vector direction); the brace code at main.js:675 correctly treats windDirection + PI as the wind source direction, confirming the field comment is inverted. Files: tests/tests.js, tests/run.html, js/main.js, js/connectome.js. [-PI-]
